# Architecture and assumptions

The simulator accepts timestamped callback jobs, releases them into a ready queue, chooses one according to the configured policy, and runs it non-preemptively. At dispatch it checks data age. At completion it checks relative deadline. This separation distinguishes stale input from slow completion.

The diagnostic is causal but intentionally local: queue interference versus execution-demand pressure. It does not claim worst-case response-time proof, multicore cache modeling, operating-system jitter, or ROS 2 executor equivalence. Those are potential adapters or separate experiments, not hidden assumptions.

Determinism comes from stable input ordinals and explicit tie breakers. For `latest-only`, each stream's sequence must increase strictly in `(release_ms, input ordinal)` order; this makes sequence the declared recency relation and rejects decreasing or repeated histories. Every recorded dispatch or supersession receives a monotonic trace ordinal. Final chronological ordering uses `(start_ms, trace ordinal)`, so simultaneous zero-duration outcomes cannot be reordered by stream names. A superseded sample remains in the trace with zero execution time. After trace order is fixed, every outcome count and timing maximum is recomputed from all trace rows, including superseded samples. Invalid and overflowing time values fail closed. The `sha256-v2:` fingerprint hashes a length-delimited canonical encoding of the trace ordinal and every input and derived trace field; the public v1 function remains available only for explicit compatibility checks.
